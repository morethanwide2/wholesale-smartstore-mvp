# Windly Benchmark Notes

이 문서는 윈들리의 사용 흐름을 벤치마킹해 우리 시스템에 반영할 기능을 정리한 기준표입니다. 목표는 화면을 복사하는 것이 아니라, 온라인 셀러가 실제로 겪는 상품 수집, 가공, 마켓별 등록, 오류 확인, 재고/주문 관리 흐름을 우리 구조에 맞게 구현하는 것입니다.

## 1. 핵심 관찰

윈들리의 핵심은 상품 데이터를 한 번에 바로 마켓으로 보내는 것이 아니라, 다음 단계를 분리해 관리하는 데 있습니다.

1. 소싱몰/도매처 상품 수집
2. 수집상품 보관
3. 등록상품으로 전환
4. 스토어별 등록정보 편집
5. 등록 전 오류 확인
6. 마켓 업로드
7. 재고 최신화
8. 주문 수집
9. 발주/배송 처리

우리 시스템도 이 흐름을 기준으로 설계해야 합니다.

## 2. 윈들리 기능 맵

| 영역 | 윈들리식 기능 | 우리 시스템 반영 |
| --- | --- | --- |
| 홈 | 처리 현황, 알림, 작업 요약 | 대시보드 필요 |
| 상품 수집 | 소싱몰 상품 가져오기 | `supplier_products` |
| 수집상품 | 원본 상품 보관/검색 | supplier detail 화면 |
| 등록상품 | 판매용 상품 편집 | `master_products` |
| 스토어별 정보 | 마켓별 상품명/카테고리/가격 | `channel_product_profiles` |
| 오류 체크 | 등록 전 필수값/금지어/카테고리 검증 | validation service |
| 재고 최신화 | 품절/재고 변경 감지 | inventory sync |
| 주문 관리 | 마켓 주문 수집/매핑 | orders, order_items |
| 발주 관리 | 발주대기/자동발주 후보 | purchase_orders |
| 설정 | 스토어 연결, 소싱몰 연결, 기본값 | env + settings tables |
| 알림 | 카카오 알림 등 | notification queue 필요 |

## 3. 반드시 벤치마킹할 UX

### 3.1 수집상품과 등록상품 분리

도매몰 원본은 수정하지 않고 보관합니다. 판매자가 실제로 수정하는 값은 등록상품 또는 채널별 등록정보에 저장합니다.

현재 반영:

- `supplier_products`
- `supplier_product_options`
- `master_products`
- `master_product_options`

추가 강화:

- 수집상품 상세 화면
- 원본 변경 이력
- 원본과 내 판매정보 비교 화면

### 3.2 마켓별 등록정보 분리

같은 상품이라도 마켓마다 필수값이 다릅니다.

- 스마트스토어: 카테고리, 옵션 구조, 고시정보, 이미지 정책
- 쿠팡: 카테고리별 필수 속성, 브랜드/제조사/인증정보
- 11번가: 상품고시, 배송정책, 옵션 제약
- 지마켓/옥션: ESM 기준 카테고리/옵션/배송정책
- 롯데온/토스쇼핑: 채널별 정책 및 필수 속성

현재 반영:

- `channel_product_profiles`
- `9_채널별_등록정보.py`

추가 강화:

- 채널별 필수 속성 테이블
- 채널별 카테고리 매핑
- 채널별 가격/배송비 정책
- 채널별 이미지/상세페이지 정책

### 3.3 등록 전 오류 체크

윈들리류 서비스의 핵심 가치는 업로드 실패를 줄이는 것입니다.

검증해야 할 항목:

- 상품명 길이
- 금지어
- 대표이미지 누락
- 상세페이지 누락
- 카테고리 누락
- 옵션명/옵션값 누락
- 옵션 구조 불일치
- 인증정보 누락
- 고시정보 누락
- 판매가/공급가 역마진
- 품절/단종 상품 등록 시도

현재 반영:

- 스마트스토어 기본 검증
- 등록 실패 큐

추가 강화:

- 검증 결과를 error/warning/info로 분리
- 오류별 해결 버튼 또는 수정 위치 안내
- 채널별 검증 엔진 분리

### 3.4 작업 대기열

API를 너무 자주 호출하면 제한에 걸릴 수 있으므로, 등록/수정/재고전송/주문수집은 모두 대기열로 관리해야 합니다.

현재 반영:

- `api_jobs`
- `api_rate_limit_settings`
- `API 전송대기열` 화면

추가 강화:

- 실제 워커
- 재시도 정책
- 실패 사유별 분류
- 중복 작업 병합

### 3.5 주문 이후 흐름

상품등록만으로 끝나면 운영 시스템이 아닙니다.

필요 흐름:

- 주문 수집
- 상품/옵션 매핑
- 미매칭 주문 검수
- 발주대기 생성
- 송장 수집
- 채널 송장 전송
- 취소/교환/반품 관리

현재 반영:

- Mock 주문 수집
- 발주대기 생성

추가 강화:

- 취소/교환/반품 테이블
- 송장 테이블 고도화
- 마켓별 주문 상태 매핑

## 4. 개발 우선순위

### Phase 1: 상품 등록 안정화

- 채널별 등록정보 화면
- 마켓별 카테고리/속성 저장 구조
- 등록 전 오류 체크 고도화
- 이미지/상세페이지/인증정보 편집 강화
- API 대기열 기본 워커

### Phase 2: 실제 API 연결

- 실제 도매처 API 연결
- 스마트스토어 실제 등록
- 스마트스토어 재고 수정
- 스마트스토어 주문 수집

### Phase 3: 멀티마켓 확장

- 쿠팡
- 지마켓/옥션
- 11번가
- 토스쇼핑
- 롯데온

### Phase 4: 운영 자동화

- 품절/단종 자동 반영
- 주문 수집 자동화
- 발주대기/발주 자동화
- 송장 전송
- 취소/교환/반품 관리

### Phase 5: 편의 기능

- 카카오 알림
- 작업 대시보드
- AI 상품명/키워드/상세설명 생성
- 대량 편집
- 엑셀 업로드/다운로드

## 5. 우리 시스템이 윈들리보다 더 좋아져야 할 부분

1. 실패 사유를 더 투명하게 보여준다.
2. 원본 상품과 판매상품의 차이를 명확히 보여준다.
3. 마켓별 등록정보를 분리해 오류를 줄인다.
4. API 호출을 대기열로 관리해 제한을 피한다.
5. 가격, 재고, 옵션 변경 이력을 남긴다.
6. 사용자가 직접 수정한 값과 도매몰에서 갱신된 값을 구분한다.
7. 자동화 전 단계에서 관리자 확인 포인트를 명확히 둔다.
